## Critical Findings

### C-1: Hardcoded private IP in ErgoPay fallback
- **File:** `api/ergopay/init/route.ts:20`
- **Category:** Deployment
- **Description:** `process.env.ERGOPAY_PUBLIC_HOST || process.env.VERCEL_URL || "192.168.110.231:3005"` — if env vars are missing in production, ErgoPay URLs point to a LAN address mobile wallets can't reach.
- **Recommendation:** Remove the `192.168` fallback. Throw if no public host is configured.

### C-2: Hardcoded Ergo node IP in 16 files
- **Files:** Every API route + `packages/core/src/config.ts`
- **Category:** Deployment / Security
- **Description:** `http://96.255.150.220:9053` is the fallback in every file that calls the node. This is a residential/dev IP over unencrypted HTTP. An MITM could serve tampered oracle data. If `ERGO_NODE_URL` is unset in production, all requests go through this address.
- **Recommendation:** Export `NODE_URL` from a single `@/lib/config.ts`. Fail loudly if env var is missing. Use HTTPS in production.

## High Findings

### H-1: Server component pages crash when node is unreachable
- **Files:** `trade/[asset]/page.tsx:65-70`, `market/page.tsx:22-26`
- **Category:** Error Handling
- **Description:** Server Components call `Promise.all([fetchSpotPrice, fetchVol, scanReserves, scanSellOrders])` with no try/catch. If the node is down, the page shows Next.js's default error page or a blank screen.
- **Recommendation:** Wrap in try/catch, render degraded UI with "Node unreachable" message.

### H-2: Height=0 on node failure misclassifies expired options
- **Files:** `reserve-scanner.ts:90,161`, `api/stats/route.ts:14`
- **Category:** Error Handling
- **Description:** When the node is unreachable, `fetchHeight()` returns 0. This propagates to classification logic where `currentHeight > maturityHeight + exerciseWindow` is false for all reserves — expired options appear as active.
- **Recommendation:** Treat height=0 as error state. Show staleness indicator.

### H-3: No client-side oracle price refresh
- **Files:** `stores/oracle-store.ts`, `trade/[asset]/write/page.tsx`
- **Category:** State Management
- **Description:** Once a page loads, price data never updates until manual refresh. A user composing an option for 30 minutes sees 30-minute-stale prices. The oracle store has setters but no auto-refresh logic.
- **Recommendation:** Add periodic client-side refresh (every 30-60s) on pages displaying prices, or re-fetch before TX submission.

### H-4: Wallet balance never updates after transactions
- **Files:** `stores/wallet-store.ts`, `WalletButton.tsx:77,113`
- **Category:** State Management
- **Description:** `ergBalance` is set once at connection and never refreshed. After writing/buying/exercising, the navbar shows the old balance.
- **Recommendation:** Refresh balance after any TX completes. Add periodic refresh.

### H-5: ErgoPay List-for-Sale Promise never resolves on modal close
- **File:** `portfolio/page.tsx:620-629`
- **Category:** Error Handling
- **Description:** `handleListForSaleSubmit` returns a Promise where `resolve` fires on `onSigned` but there's no `reject` path. If the user closes the modal or the request expires, the Promise never settles — `setSubmitting(true)` is never reset, leaving the button permanently disabled.
- **Recommendation:** Add `onExpired`/`onClose` handlers that call `reject()`.

### H-6: Wallet disconnect mid-flow produces cryptic errors
- **File:** `TradePanel.tsx:249`
- **Category:** Error Handling
- **Description:** If Nautilus is closed during TX signing, `api.sign_tx()` throws a generic TypeError. Error handling checks for "declined"/"Refused" but doesn't catch dead wallet connections. User sees truncated "Error: Cannot read prop..."
- **Recommendation:** Wrap signing with connectivity check. Catch TypeError and show "Wallet disconnected — please reconnect."

### H-7: Option chain / Market / Portfolio tables unusable on mobile
- **Files:** `OptionChain.tsx`, `MarketFilters.tsx`, `portfolio/page.tsx`
- **Category:** Mobile UX
- **Viewport:** Mobile
- **Description:** Option chain has 11 columns, market table has 7, portfolio tables have 7-8. All rely on `overflow-x-auto` with no scroll indicator. The option chain is the core trading interface and is practically unusable on phones. Market table doesn't even have `overflow-x-auto`. Action buttons have tiny touch targets (`text-xs px-2 py-1`).
- **Recommendation:** Design mobile-specific layouts: card-based per strike (option chain), column hiding (market), or tabbed views. Increase touch targets to 44px minimum.

### H-8: Color contrast fails WCAG AA
- **Files:** `globals.css`, all components
- **Category:** Accessibility
- **Description:** Several text/background combinations fail WCAG AA (4.5:1):
  - `--text-dim` (#505870) on `--background` (#0a0c10): ~2.9:1
  - `--text-secondary` (#8891a5) on `--background` (#0a0c10): ~4.3:1
  - `--accent-copper` (#c87941) on `--background` (#0a0c10): ~4.0:1
  - Disabled text at 30% opacity is extremely low contrast
- **Recommendation:** Lighten `--text-dim` to ≥#7a82a0, `--text-secondary` to ≥#9da5b8. Review all `/30` and `/40` opacity text.

## Medium Findings

### M-1: No input validation on API route parameters
- **Files:** `box-by-token/route.ts:9`, `address-to-raw/route.ts:9`, `mempool/[txId]/route.ts:10`
- **Category:** Security
- **Description:** `tokenId`, `address`, and `txId` params are checked for presence but not format. Malicious values with path-traversal chars could enable SSRF against the node API. Compare `box-by-id` which validates `^[0-9a-f]{64}$`.
- **Recommendation:** Validate `tokenId` as `^[0-9a-f]{64}$/i`, `address` as `^[1-9A-HJ-NP-Za-km-z]{30,60}$`, `txId` as `^[0-9a-f]{64}$/i`.

### M-2: No rate limiting on any API route
- **Files:** All routes under `api/`
- **Category:** Security
- **Description:** `/api/submit` proxies to the node's `/transactions` and could be used to flood it. `/api/reserves` and `/api/my-boxes` make multiple node requests per call, amplifying DoS potential.
- **Recommendation:** Add rate limiting (e.g., `@upstash/ratelimit`). At minimum rate-limit `/api/submit`.

### M-3: Supabase credentials hardcoded as fallback
- **File:** `lib/supabase.ts:3-4`
- **Category:** Deployment
- **Description:** Supabase URL and anon key hardcoded. While anon keys are designed for client use, they can't be rotated without a code change.
- **Recommendation:** Require env vars, fail at build time if missing.

### M-4: `next.config.mjs` is empty — no security headers
- **File:** `next.config.mjs`
- **Category:** Deployment
- **Description:** No `poweredByHeader: false`, no CSP, no X-Frame-Options, no image optimization config.
- **Recommendation:** Add security headers, `poweredByHeader: false`, consider `output: 'standalone'`.

### M-5: Google Fonts loaded via render-blocking `<link>`
- **File:** `app/layout.tsx:30-35`
- **Category:** Performance
- **Description:** Three Google Font families via `<link rel="stylesheet">` block first paint. Geist fonts already use `next/font/local` correctly.
- **Recommendation:** Migrate to `next/font/google` for self-hosting and `font-display: swap`.

### M-6: Duplicate companion box fetches
- **File:** `lib/oracle-parser.ts:18-57,64-123`
- **Category:** Performance
- **Description:** `fetchSpotPriceByIndex()` and `fetchVolByIndex()` each fetch the full companion box independently. Called in `Promise.all` on the trade page = two identical HTTP requests.
- **Recommendation:** Create a single `fetchCompanionBox()` returning both R8 (prices) and R5 (vol).

### M-7: Zustand stores subscribed without selectors
- **Files:** `portfolio/page.tsx:236`, `WalletButton.tsx:18`, `TradePanel.tsx:50`
- **Category:** Performance
- **Description:** Components destructure from the store hook without selectors, causing re-renders on any store change, including unrelated fields.
- **Recommendation:** Use shallow selectors: `useWalletStore(useShallow(s => ({ connected: s.connected })))`.

### M-8: 23 console.log statements in production code
- **Files:** `portfolio/page.tsx` (17), `TradePanel.tsx` (1), `write/page.tsx` (1), `ergopay/reduce/route.ts` (1), `ergopay/callback/route.ts` (2), `ergopay-adapter.ts` (1)
- **Category:** Code Quality
- **Description:** Debug logging in production. Portfolio page alone has 17 occurrences. ErgoPay route logs up to 2KB of TX payload including addresses.
- **Recommendation:** Replace with structured logger gated behind `NODE_ENV === 'development'`.

### M-9: ASSET_MAP defined in 3 places
- **Files:** `lib/asset-map.ts` (canonical), `trade/[asset]/page.tsx:14-38`, `app/page.tsx:12-56`
- **Category:** Code Quality
- **Description:** Shared `asset-map.ts` exists but trade page and dashboard define their own copies. Wizard correctly imports from shared module.
- **Recommendation:** Import from `@/lib/asset-map` everywhere. Derive `CATEGORIES` from the shared map.

### M-10: ErgoPay in-memory session store not production-ready
- **File:** `lib/ergopay.ts:23-27`
- **Category:** Architecture
- **Description:** Uses `globalThis` Map. In Vercel with multiple serverless instances, sessions aren't shared. Cleanup only runs when new requests arrive. Code acknowledges "In production, use Redis or DB."
- **Recommendation:** Implement Redis/DB-backed storage before launch.

### M-11: ErrorBoundary only covers `/app/` routes
- **Files:** `app/layout.tsx` (root — no boundary), `app/app/layout.tsx:27` (app — has boundary)
- **Category:** Error Handling
- **Description:** Landing, learn, and how-it-works pages are unprotected. Also, React ErrorBoundary only catches render errors, not async/event-handler errors (where most DeFi failures occur).
- **Recommendation:** Add ErrorBoundary to root layout. Consider global `unhandledrejection` handler.

### M-12: Write page shows no loading state for oracle data
- **File:** `trade/[asset]/write/page.tsx:139-173`
- **Category:** UX
- **Description:** Fetches oracle data in useEffect with no loading indicator. Form is interactive with `spotPrice=0` until fetch completes. User could fill the form with stale defaults, then have values overwritten.
- **Recommendation:** Show skeleton/spinner, disable form until oracle data loads.

### M-13: `handleListForSaleClick` blocks ErgoPay users
- **File:** `portfolio/page.tsx:474`
- **Category:** State Management (Bug)
- **Description:** Guard `if (!api) return;` silently blocks the action for ErgoPay users (where `connected=true` but `api=null`).
- **Recommendation:** Change to `if (!api && walletType !== "ergopay") return;`.

### M-14: No active-state indicator on navbar links
- **File:** `Navbar.tsx`
- **Category:** UX
- **Viewport:** Both
- **Description:** No `usePathname` import, no active styling on nav links. Every link looks the same regardless of current page. No `aria-current="page"`.
- **Recommendation:** Apply distinct style + `aria-current="page"` to the active link.

### M-15: WizardBanner has no dismiss mechanism

- **Description:** Appears on every `/app/*` page with no close button and no localStorage persistence. Returning users see it every time.

### M-16: Sell tab clickable but permanently disabled

### M-17: Expiry tabs always empty

### M-24: No skip-navigation link + no prefers-reduced-motion
- **Files:** `app/app/layout.tsx`, `globals.css`

### M-25: ErgoPayModal + WalletButton QR modal lack focus trap / ARIA

- **Description:** ErgoPayModal has `role="dialog"` but doesn't use `useFocusTrap`. WalletButton QR modal uses inline styles and has no ARIA attributes at all. Both allow focus to escape behind the overlay.
- **Recommendation:** Add `useFocusTrap` to ErgoPayModal. Refactor WalletButton QR modal to use proper ARIA + focus trap.
